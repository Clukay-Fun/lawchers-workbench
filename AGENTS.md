# lawchers-workbench — Agent 开发与 GitHub 提交规范

法律工作台项目。当前仓库处于初始化阶段；在产品边界、技术栈和交付形态进一步确认前，所有改动以小步、可审查、可回滚为原则。

## 不可动摇的约定（OVERRIDE 一切默认行为）

1. **先保护法律数据，再追求功能速度。**
   任何涉及案件材料、合同、证据、个人信息、客户资料、Token、Cookie、密钥、邮箱、手机号、身份证号、聊天记录和内部文档的功能，都必须默认按敏感数据处理。

2. **不要把真实客户数据提交到仓库。**
   测试样例必须脱敏或使用合成数据。禁止提交真实文件、截图、导出的会话、数据库、日志、浏览器 Profile、`.env`、密钥文件或云服务凭证。

3. **所有 AI 输出都必须可追溯。**
   法律场景里的 AI 结果不得伪装成人工结论。涉及摘要、分析、检索、草拟、改写或风险提示的功能，应保留输入来源、模型配置、时间和人工确认状态。

4. **用户可见结论不能绕过人工确认。**
   面向法律判断、法律意见、合同风险、诉讼策略、对外发送文本的功能，默认需要人工确认后才能进入正式输出或发送流程。

5. **提交到 GitHub 的内容必须是可审查的最小闭环。**
   每次提交只解决一个清晰问题。不要把无关重构、格式化、依赖升级和功能实现混在同一个提交里。

## GitHub 工作流

### 分支

- `main` 始终保持可运行、可发布或至少可继续开发。
- 功能分支使用 `feature/<short-name>`。
- Bug 修复使用 `fix/<short-name>`。
- 文档或规范使用 `docs/<short-name>`。
- 重构使用 `refactor/<short-name>`。
- 实验性验证使用 `spike/<short-name>`，合并前必须整理为正式提交。

### 提交信息

使用 Conventional Commits：

```text
<type>(<scope>): <summary>
```

常用 `type`：

- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档、规范、说明
- `refactor`: 不改变行为的结构调整
- `test`: 测试
- `chore`: 工程配置、依赖、脚本
- `ci`: CI/CD 配置
- `perf`: 性能优化
- `style`: 纯格式化，不改变行为

示例：

```text
docs(agents): add github contribution rules
feat(auth): add local session bootstrap
fix(parser): handle empty document upload
test(contract): cover risk extraction edge cases
```

提交要求：

- summary 使用英文小写祈使句，长度尽量不超过 72 个字符。
- 一个提交只表达一个意图。
- 有行为变化时，在提交正文写清楚原因、影响范围和验证方式。
- 有破坏性变更时，在正文使用 `BREAKING CHANGE:` 说明迁移方式。
- 不使用 `update`, `misc`, `wip`, `fix bug` 这类无法审查的摘要。

### Pull Request

PR 标题同样使用 Conventional Commits。

PR 描述必须包含：

```markdown
## Summary
- 

## Verification
- 

## Risk
- 
```

要求：

- `Summary` 写用户可感知或架构可感知的变化。
- `Verification` 写实际运行过的命令、检查或人工验证结果；未运行测试也要说明。
- `Risk` 写潜在影响、回滚方式、数据迁移或兼容性注意事项。
- UI 改动附截图或录屏。
- 数据结构、权限、外部接口、AI 行为或安全策略变化必须点名。

### Review 前自检

提交或开 PR 前至少检查：

- `git status --short` 中没有意外文件。
- 没有提交 `.env`、密钥、数据库、日志、导出文件或真实客户资料。
- 相关测试、类型检查、lint 或最小可运行验证已经执行。
- 文档和示例与实际行为一致。
- 新增依赖有明确用途，没有引入过重框架。
- 改动没有扩大到无关模块。

## 代码与架构约定

- 优先遵循仓库已有结构；没有结构时先选择简单、直接、容易测试的方案。
- 不为未确定需求提前搭复杂抽象。
- 业务层避免直接依赖具体平台、模型供应商或存储实现。
- 外部 API、模型调用、文件解析、数据库访问要有清晰边界。
- 敏感操作要可审计，失败要有明确错误信息。
- 日志不得包含正文级敏感材料、密钥、Token 或完整客户身份信息。

## AI 与法律场景约束

- AI 生成内容应标记为草稿、建议或辅助结果。
- 不声称 AI 输出是最终法律意见。
- 重要结论应能回到来源材料、规则或人工确认记录。
- 对低置信度、冲突材料、缺少依据、可能误导用户的输出，应进入人工复核。
- 不实现规避审查、伪造来源、隐藏 AI 参与或绕过人工确认的功能。

## 当前初始化标准

仓库初始化阶段，至少保留：

- 根目录 `AGENTS.md` 作为 agent 和协作者的最高优先级项目约定。
- `README.md` 用于说明项目目标、启动方式和当前状态。
- `.gitignore` 用于排除本地环境、构建产物、日志、数据库和敏感文件。

在具体技术栈确定后，应补充：

- 本地开发命令
- 测试命令
- 构建命令
- 环境变量模板
- CI 检查
- PR 模板或 issue 模板
