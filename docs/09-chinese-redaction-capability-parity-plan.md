# 开发计划 09 · 中文法律材料脱敏能力追平/超过 legal-anonymizer

> 主题：把**中文**法律材料的脱敏能力做到追平、并力求超过 legal-anonymizer。
> 主战场：**引擎仓库 legal-desensitizer**（类型/检测/输出/保真都在引擎）；工作台只消费暴露。
> 与 docs/08 区分：08 是"只读诊断面板"，本计划是"能力路线"，两者不混。

## 范围红线（明确不做）
- 不做英文脱敏。
- 不做英文 LLM。
- 不做 Ollama。
- 不做涉外英文补盲。
- 一切只针对中文法律材料。

## 当前基线（避免重复造轮子）
- 引擎现有约 **17 条正则规则**（手机/座机/身份证/护照/邮箱/案号/执行依据/统一社会信用代码/银行卡/银行信息/金额/车牌/不动产权证/密钥/日期(默认关)/银行账号(默认关)）+ **NER**（PERSON/ORG/ADDRESS(LOC)/TIME）。
- 已具备：docx 内容级可逆脱敏（body/表格/页眉页脚/批注）、文本 PDF 真删除（切片 2 已做）、扫描 PDF 像素级（redact-scan）、md parse。
- 本计划在此基线上**补齐到 30+ 类型、规整双层检测、统一三格式输出、增强保真**。

---

## P0 · 30+ 中文敏感类型覆盖
- 在现有 ~17 类基础上扩到 **30+ 中文敏感类型**（按法律材料常见项补：如各类证件号、社保/公积金、车架号、邮编、IP、未成年人信息、医疗信息、亲属关系称谓+人名等——具体清单先列再实现）。
- **验收**：
  - 每类敏感类型有**合成样本**（不得用真实数据）。
  - 每类有**检测测试**。
  - 每类有 **mask / audit 验证**（掩码正确 + 审计可追溯）。

## P1 · 正则 + 中文 NER 双层检测（分层规整）
- regex 层只管结构化确定性数据；NER 层管姓名/机构/地址等语义实体；两层职责清晰、不互相污染（参考 legal-anonymizer 的结构化隔离思路）。
- **验收**：
  - regex 与 NER 分层清楚（可分别开关、可分别审计）。
  - NER 不可用时**明确降级**（不静默，审计/输出标注 regex-only）。
  - NER 可用时**姓名/机构/地址召回提升**（给出前后对比数据）。

## P2 · 一次生成 MD / DOCX / PDF 三种输出
- 同一材料一次操作产出 Markdown / DOCX / PDF 三种脱敏输出（对标 legal-anonymizer 的 triple output）。
- **验收**：
  - 同一材料导出 md / docx / pdf 三份。
  - map / audit / decision log 一并导出。
  - 三份均通过**残留审计**。

## P3 · DOCX / PDF 保留格式增强
- **DOCX**：跨 run / 表格 / 页眉页脚（部分已覆盖，补齐缺口并测试）。
- **文本 PDF**：真删除（切片 2 已做，纳入回归）。
- **扫描 PDF**：像素级脱敏（redact-scan，纳入回归 + 坐标质量）。
- **验收**：
  - DOCX 跨 run/表格/页眉页脚均正确脱敏且格式不坏。
  - 文本 PDF 真删除（原文不可提取）。
  - 扫描 PDF 像素级擦除 + 残留验证通过。

---

## 执行约束
- 分阶段、分提交：P0 → P1 → P2 → P3，每个 P 自身可验收后再下一个。
- 主要在 legal-desensitizer 仓库；涉及工作台暴露的部分单独提交、单独说明。
- 验证铁律：每类/每格式贴真实运行输出（样本、检测结果、map/audit、残留）；禁止只跑测试名/build-lint 充验证；不得用真实客户数据。
- 不碰范围红线里的英文/LLM/Ollama。
- Conventional Commits、原子提交、当前分支、不 push。
