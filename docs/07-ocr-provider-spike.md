# 开发计划 07 · OCR 引擎评估 Spike（RapidOCR vs PP-OCRv6）

> 这是**评估性 spike**，不是上线功能。产出 = 证据 + 对比表 + 一句话结论。
> 要回答的一句话：
>
> **「PP-OCRv6 是否值得替换 RapidOCR 成为扫描 PDF 的默认 OCR 引擎？」**
>
> 评估重点不是"谁全文识别更准"，而是**谁更适合扫描 PDF 脱敏闭环**（召回 + 坐标 + 安装 + 稳定）。
> 仓库：主要在 `legal-desensitizer`（`redact-scan` 当前用 RapidOCR）；工作台基本不动。

---

## 第 0 步 · 先核实，别假设（硬前置）
- **确认 `PP-OCRv6` 是真实发布版本**（PaddleOCR 公开线常见到 PP-OCRv4；v6 是否存在/叫法是否正确要先查证）。若不存在，就用**当前可装到的最新 PaddleOCR/PP-OCR 版本**，并在报告里写明实际用的版本号。
- **确认 macOS arm64 能装**（PaddlePaddle/PaddleOCR 的 arm64 wheel）。装不上 → 直接记录"安装不可行"，评估终止于安装维度（准确率再高也不能做默认）。

## 样本（不得用真实客户数据）
- 用**合成/脱敏扫描件**：复用 `legal-desensitizer/tests/gen_scan_fixtures.py` 与现有 scan 样本，或把合成文本 PDF 栅格化成图片 PDF 模拟扫描件。
- 覆盖：劳动合同 / 解除通知 / 仲裁申请；敏感实体密集（姓名/手机/身份证/公司/金额/日期/地址）；低清晰度；旋转/歪斜；表格型。
- **最小 spike 必跑**：同一份 **3 页扫描 PDF**，两个 provider 各跑一遍。

---

## 评估维度

### 1. 安装成本（一票否决项）
RapidOCR / PP-OCRvX 各记：pip 安装是否稳定、macOS arm64 wheel 是否顺、模型下载大小、是否可离线缓存、对**双击安装体验**的影响、安装耗时。

### 2. 敏感实体召回（不是全文准确率）
同批样本，按实体类型看：姓名 / 手机号 / 身份证号 / 公司名 / 金额 / 日期 / 地址。
- **实体召回率** = 应识别敏感实体中，OCR 文本能被规则/NER 命中的比例。
- **误识别率** = OCR 错字导致规则误判/漏判的比例。

### 3. 坐标质量（扫描件脱敏最关键）
链路：`OCR 文本 → 敏感实体字符区间 → 页面 polygon/box → 像素擦除`。看：是否有 line / word / char box；box 是否稳定完整覆盖文字；斜体/旋转覆盖是否准；**同页重复文字能否只擦指定位置**。
- 只给 line box → 能用但精确擦除偏粗；给 word/char box → 优势大。

### 4. 速度
单页耗时 / 10 页耗时 / CPU 占用 / 内存峰值 / 是否需要 GPU。**普通用户机器不能默认要求 GPU**。

### 5. 输出稳定性
同一文件跑 3 次：文本是否一致、box 是否一致、候选数量是否一致。法律脱敏最怕"这次识别出、下次识别不出"。

### 6. 脱敏闭环验证（最终判据，非 OCR demo）
每个 provider 跑完整链路：
```
扫描 PDF → OCR → prepare manifest/candidates → 人工确认/取消
→ 导出像素级脱敏 PDF → 再 OCR / 文本残留扫描 → audit passed
```

## 结论分支（spike 要给出落点，不要含糊）
- **PP-OCRvX 准确率明显更好 + 安装可控 + 坐标足够细** → 替换默认。
- **准确率更好但安装重** → 默认 RapidOCR，PP-OCRvX 作"高级模式"可选。
- **坐标不适合精确擦除** → PP-OCRvX 最多用于"扫描件转 Markdown 预览"，**不用于正式脱敏导出**。
