# 开发计划 14 · P5 扫描件 OCR 敏感候选接线 + 文档区工具条 + 横向进度条

> 接续 docs/13（P4）。P4 修对了"遮蔽框≠OCR框"，但还差三条线没收干净：扫描件自动脱敏接线、文档区工具条位置、进度条形态。
> **关键事实（已核）**：引擎已能产出**带 `entityType` 的 OCR 候选**（`legal-desens analyze` 的 `ocrBoxes[].entityType`、prepare 的 candidates）。所以扫描件自动脱敏**主要是工作台接线**，不是缺 ML 能力。
> 范围：**不做公章算法增强**（另开 P5.5/P6）；不引大框架。

## 两条管线彻底分开（这版核心认知）
- **遮蔽模式**：用**视觉框**。遮蔽框 = 带类型的候选（命中敏感类型的 OCR 候选）+ 公章候选 + 强制脱敏词 + 用户手动框。
- **星号/占位模式**：用**文本实体**，**不依赖遮蔽框**。来源 = OCR 全文 + 规则/强制词/可用 NER 命中的实体。

> 链路应是：`OCR 框 → 规则/NER/强制词匹配 → 命中的敏感候选 → 遮蔽框`；不是"OCR 框全部遮蔽"，也不是"OCR 框全丢、只能手动"。

---

## 六项改动（每项带验证）

1. **横向进度条**（替换当前四行步骤文字）
   - 上传：真实百分比（XHR upload progress）→ `上传中 42% [████░░░]`。
   - 处理：无真实百分比时用 **indeterminate 横条** + 当前步骤文案（正在 OCR 识别…），**不伪造百分比**。
   - verify：上传大文件看到真实%横条；处理阶段是滚动横条不是假数字。

2. **文档区局部 sticky 工具条移位**（按用户图中红框）
   - 工具条移到**文档滚动区内部、双栏页面上方**，随文档区 sticky：模式（遮蔽/星号/占位）/ 页码 `1/7` 跳转 / 保存框 / 换文件。
   - **导出按钮仍留全局右上角**，不塞进这个工具条。
   - verify：滚动时工具条在文档区顶部吸顶；导出在右上独立。

3. **扫描件自动脱敏接线（遮蔽模式）**
   - 把引擎已产出的**带 entityType 的 OCR 候选**接成遮蔽框：只有命中敏感类型/强制词的候选才进遮蔽框；裸 OCR 框不进。
   - verify：上传扫描件，遮蔽模式**自动**黑掉敏感命中（手机号/身份证/姓名/机构等），不再"只能手动"、也不全页黑。

4. **星号/占位改为基于 OCR 文本实体**（不依赖遮蔽框）
   - 文本双栏：左 = OCR 全文（命中高亮）；右 = 替换后文本预览。来源 = OCR 文本 + 检测实体，与视觉遮蔽框解耦。
   - verify：扫描件切到星号/占位，右侧出现**自动替换**后的文本（星号/占位），不再空着。

5. **星号/占位扫描件导出 TXT/MD/DOCX 自动脱敏**
   - 导出走文本管线，按检测实体替换；扫描件不回写 PDF。
   - verify：扫描件星号/占位各导出 TXT/MD/DOCX，敏感实体**已自动替换**（贴导出文件）。

6. **手动框仅影响遮蔽模式**
   - 手动框只进遮蔽框；"手动框 → 文本模式（框内 OCR 反查）"**后置**，本轮不做。
   - verify：手动框在遮蔽模式生效；切到文本模式不依赖手动框。

## 不做
- 公章识别率增强（P5.5/P6）。
- 手动框反查 OCR 文本喂给文本模式（后置）。
- 大框架/引擎 ML 改造。

> 若接线时发现引擎对某类扫描候选**未给 entityType / 未跑检测**，那一小块是引擎补检测的 follow-up（legal-desensitizer），**先停下确认**，不在工作台硬凑。
